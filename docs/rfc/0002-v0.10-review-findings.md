# RFC 0002 — v0.10.0 Internal Review Findings

Status: **active** · Target: v0.10.0 cleanup / v0.10.1 · Date: 2026-05-21

Internal review of the 14 plugin packages after RFC 0001 phases 1–9
shipped. Audits consistency, cohesion, DRY, testing, patterns.

## 🟥 High-priority (block-or-fix before v0.10.0 npm publish)

### F1. Massive DRY violation across the 4 OpenAI-shaped adapters

`GroqApiAdapter`, `CerebrasApiAdapter`, `MistralApiAdapter`,
`OpenRouterApiAdapter` share ~95% of identical wire code. After
stripping comments, the diff between any two is ~30 lines. The
real differences:

| Field | Groq | Cerebras | Mistral | OpenRouter |
|---|---|---|---|---|
| Endpoint | `api.groq.com/openai/v1` | `api.cerebras.ai/v1` | `api.mistral.ai/v1` | `openrouter.ai/api/v1` |
| Token field | `max_completion_tokens` | `max_completion_tokens` | `max_tokens` | `max_tokens` |
| Default model | `llama-3.3-70b-versatile` | `gpt-oss-120b` | `mistral-small-latest` | `meta-llama/llama-3.3-70b-instruct:free` |
| Extra headers | — | — | — | `HTTP-Referer`, `X-Title` |
| Tool fallback | none | try/catch → plain chat | none | none |

**Fix**: Ship `OpenAiCompatibleAdapter` abstract base in
`@noocodex/dagonizer/adapter`. Each provider becomes 30 lines of
configuration:

```ts
class GroqApiAdapter extends OpenAiCompatibleAdapter {
  constructor(opts) {
    super({
      ...opts,
      endpoint:       'https://api.groq.com/openai/v1/chat/completions',
      defaultModel:   'llama-3.3-70b-versatile',
      tokenField:     'max_completion_tokens',
      extraHeaders:   {},
      capabilities:   { toolUse: 'full', structuredOutput: true, jsonMode: true },
    });
  }
}
```

Removes ~600 LOC of duplication and makes adding a new OpenAI-compatible provider a ~10-line task.

### F2. Per-package READMEs are placeholders

Every plugin has a 6-line stub README. Consumers landing on
`npmjs.com/package/@noocodex/dagonizer-adapter-groq` deserve real
documentation: install, usage example, configuration options,
provider-specific caveats (rate limits, model selection, capabilities).

**Fix**: Write proper mini-docs for each of the 14 plugins. **Done in
this review pass.**

### F3. Pattern packages ship only the parent classes; named leaves missing

RFC 0001 Phase 7 promised: `ClassifyIntentNode`, `DecideToolsNode`,
`ValidateResponseNode`, `RankCandidatesNode`, `ComposeResponseNode`,
`ComposeEmptyResponseNode`, `ComposeMemoryResponseNode`, `DeclineNode`,
`PickByScoreNode`, `SortByNode`, `DedupeByKeyNode`, `GroupByFieldNode`,
`FanInReducerNode`, `RecallContextNode`, `RecordFindingsNode`,
`MemoryDigestNode`.

Currently shipped: `DecisionNode`, `ComposeNode`, `ScoutNode`,
`GraphNode`, `RecallContextNode`, `RecordFindingsNode`,
`MemoryDigestNode`, `FlowNode`, `SelectNode`, `PickByScoreNode`,
`SortByNode`, `ReduceNode`, `DedupeByKeyNode`, `GroupByFieldNode`,
`FanInReducerNode`, `PredicateGateNode`, `ExtractFieldNode`,
`RespondNode`.

Missing the four `DecisionNode` leaves + four `ComposeNode` leaves.
Without them, consumers extend the parents and reinvent the narrowing
the leaves were supposed to provide.

**Fix**: Ship the eight missing leaves. **Done in this review pass.**

## 🟨 Medium-priority

### F4. Book / Candidate entity types duplicated across 3 tool packages

`packages/dagonizer-tool-openlibrary/src/entities.ts`,
`packages/dagonizer-tool-googlebooks/src/entities.ts`, and
`packages/dagonizer-tool-wikipedia/src/entities.ts` are identical
(50 LOC each). `CanonicalId.ts` lives only in `openlibrary` but is
the canonical dedupe utility.

**Options:**
- (a) Consolidate in `openlibrary`; other tools depend on it for
  the shared Book/Candidate types and CanonicalId.
- (b) Hoist into a tiny `@noocodex/dagonizer-book-entities` package.
- (c) Accept the duplication as the cost of independent versioning.

Recommend (a) — `openlibrary` is the "canonical" book source, has
the most metadata coverage, and the dep direction is natural.

### F5. DecisionNode + ComposeNode share buildRequest()

Both have identical `buildRequest(prompt, signal)` methods. Should
be a static helper on a shared abstract `LlmDispatchNode` or just a
free function exported from the package.

### F6. No per-package unit tests

Every plugin has `"test": "echo 'no unit tests'"`. The transport
smoke suite (workspace-level) covers wire format for the four cloud
adapters but doesn't isolate per-package behavior.

**Fix**: Each plugin should have at least one isolation test:
- Adapter packages: instantiate with dummy key, assert capability shape
- Tool packages: mock HttpTransport, assert input normalization
- Pattern packages: instantiate a minimal subclass, run execute() against a mock service

## 🟩 Low-priority / polish

### F7. Inconsistent adapter constructor option types

Some adapter `*Options` interfaces have only `apiKey`; some add `model`
+ `maxAttempts`. No canonical option type.

### F8. tsconfig.json files identical across all 14 plugins

Could hoist into `tsconfig.package.json` at the root. Saves 5 lines
per package, ~70 LOC total.

### F9. `"lint": "echo 'lint via workspace root'"` on every plugin

The workspace lint runs from the dagonizer package's eslint config.
Each plugin technically isn't lint-checked in isolation, just via
the dagonizer package's eslint . sweep. Either set up per-package
lint or accept the workspace-level coverage.
