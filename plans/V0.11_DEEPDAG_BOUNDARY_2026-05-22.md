# v0.11 — DeepDAG Boundary

Two related additions for `@noocodex/dagonizer`. Both address how data crosses
the deep-DAG boundary; together they make sub-DAGs first-class function-call
units while preserving the engine's isolation and resume guarantees.

## Scope

1. **Typed `stateMapping`** on `DAGBuilder.deepDAG()`. Same wire shape; better
   types at authoring time.
2. **`Store` contract + `BaseStore` abstract class + `MemoryStore` reference
   implementation.** Adapter pattern mirroring `LlmAdapter` / `BaseAdapter`.
   Lives in the services bag; shared mutable state without breaking sub-DAG
   isolation.
3. **`Checkpoint` integration** for named stores. Resume captures shared state
   alongside parent-state snapshots.
4. **`docs/guide/shared-state.md`** decision-matrix guide.

Plugin packages (`dagonizer-store-sqlite`, `dagonizer-store-rdf`, etc.) are
deferred to follow-on releases. v0.11 ships the contract, the base class, the
reference implementation, and the checkpoint hook. Plugin authors write
against the contract.

## Non-scope

- `Path<TParentState>` typed dotted paths on the parent side of stateMapping
  (parent paths stay `string`; child keys narrow).
- Schema-typed `Store<Schema>` (single declared key→value mapping).
- Distributed / remote stores. Contract is async-throughout so it remains
  forward-compatible.
- `DAGDeriverAnnotations` typed variant. Builder gains the type-safe form;
  derive keeps the string form until a consumer needs the typed variant.

## Architectural decisions

### D1 — Typed `stateMapping`

- Generic on the call site: `.deepDAG<TChildState>('name', 'dag-name', { … })`.
- `inputs: { [K in keyof TChildState]?: string }` — child key strict, parent
  path stringly-typed.
- `outputs: { [K in keyof TParentState]?: string }` — parent key strict where
  the parent state type is in scope (default `NodeStateBase` keys), child
  path stringly-typed.
- JSON wire format unchanged. The narrowing is purely a builder concern.
- Default generic is `NodeStateBase`, preserving every existing call site.

### D2 — `Store` contract

`src/contracts/Store.ts`:

```ts
export interface Store<TValue = unknown> {
  get<T extends TValue = TValue>(key: string): Promise<T | undefined>;
  set<T extends TValue = TValue>(key: string, value: T): Promise<void>;
  has(key: string): Promise<boolean>;
  delete(key: string): Promise<boolean>;
  update<T extends TValue = TValue>(key: string, fn: (current: T | undefined) => T): Promise<T>;
  snapshot(): Promise<StoreSnapshot>;
  restore(snapshot: StoreSnapshot): Promise<void>;
}
```

- **Async throughout** so durable backings (SQLite, network) are first-class.
  In-memory implementations resolve synchronously under the hood.
- **`update(key, fn)` is atomic within a single store instance.** Callback is
  permitted here as a deliberate carve-out from the engine's "zero callbacks"
  rule: store-level composition, not topology behavior. Documented in JSDoc.
- **`set + get` is not atomic.** Race conditions are the consumer's problem;
  `update` exists to make atomicity easy.
- **`snapshot()` returns a typed envelope** with `version`, `type`, and
  `entries`. Restore validates the envelope.

### D3 — `BaseStore` abstract class

`src/store/BaseStore.ts`:

- Implements `Store` against `protected abstract` hooks the subclass fills.
- Owns: snapshot envelope (`{ version, type, entries }`), default `update`
  implementation (read-modify-write atop `performGet` + `performSet`), key
  validation, optional namespace prefix, lifecycle no-ops
  (`connect()` / `disconnect()`), error wrapping into `StoreError`.
- Abstract methods:
  - `protected abstract performGet<T>(key): Promise<T | undefined>`
  - `protected abstract performSet<T>(key, value): Promise<void>`
  - `protected abstract performHas(key): Promise<boolean>`
  - `protected abstract performDelete(key): Promise<boolean>`
  - `protected abstract performSnapshotEntries(): Promise<readonly [string, JsonValue][]>`
  - `protected abstract performRestoreEntries(entries): Promise<void>`
- Default `update`: subclasses with native CAS override; the default works
  correctly under single-threaded JS for in-memory stores and serves as a
  fallback for stores without transactions.

Modeled directly on `BaseAdapter` in `src/adapter/BaseAdapter.ts`. Same shape:
contract interface + abstract base + protected hooks.

### D4 — `MemoryStore` reference implementation

`src/store/MemoryStore.ts`:

- `extends BaseStore<TValue>`.
- Backing: `Map<string, TValue>`.
- Synchronous read-modify-write inside async methods (returns resolved
  promises).
- 50-line subclass; most logic inherited from `BaseStore`.

### D5 — Checkpoint integration

`src/checkpoint/Checkpoint.ts` extends with named-store snapshots:

```ts
const checkpoint = await Checkpoint.from(state, {
  stores: { memory: memoryStore, audit: auditStore },
});

await checkpointStore.save(runId, checkpoint);

// Resume:
const recalled = await checkpointStore.load(runId);
const memory = new MemoryStore();
const audit  = new MemoryStore();
await recalled.restoreStores({ memory, audit });
const state = PipelineState.restore(recalled.stateSnapshot);
```

- `CheckpointDataSchema` gains `stores?: Record<string, StoreSnapshot>` —
  optional in the schema, defaulted to `{}` on restore. Forward-compatible:
  old checkpoints load with empty stores.
- `Checkpoint.from(state, { stores })` collects snapshots in parallel.
- `RecalledCheckpoint.restoreStores({ … })` accepts a map of store instances
  keyed by the same names; missing keys throw (loud failure beats silent
  desync).
- Schema-version field on `StoreSnapshot` makes adapter-side migration the
  plugin's responsibility.

### D6 — Subpath export

`@noocodex/dagonizer/store` — re-exports `Store`, `BaseStore`, `MemoryStore`,
`StoreSnapshot`, `StoreError`. Mirrors `/adapter` and `/runtime`.

Root barrel (`src/index.ts`) also re-exports these for ergonomics.

## Phasing

Five implementation phases, gated by typecheck + lint + tests at each step.
Phases marked `‖` can run in parallel; `⇣` are sequential.

```
Phase 0 ── plan doc + branch + changeset stub (this commit)
   ⇣
Phase 1 ‖ Phase 2  (independent)
   │
   ├─ Phase 1: typed stateMapping on DAGBuilder
   │
   └─ Phase 2: Store contract + BaseStore + MemoryStore
       ⇣
      Phase 3: Checkpoint integration (depends on Phase 2)
       ⇣
      Phase 4: docs/guide/shared-state.md + builder/deepflows updates
```

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `BaseStore`'s abstract surface is the de facto contract for plugin authors; changing it later breaks every plugin | Lock the protected method set by writing a minimum-viable test plugin (in-tests, not published) that exercises every override. If the plugin compiles and round-trips snapshot/restore using only the documented overrides, the surface holds. |
| Checkpoint schema additive change breaks v0.10 readers consuming v0.11 checkpoints | Document that checkpoint format moves with minor versions; consumers shouldn't load checkpoints across version gaps without testing. `StoreSnapshot.version` field gives adapters a per-plugin migration hook. |
| `update`'s callback reads as inconsistent with "zero callbacks" rule | JSDoc carves it out explicitly: callbacks are forbidden in topology; permitted for in-process atomicity primitives. Same logic that makes `NodeInterface.execute()` a function. |
| Consumer race conditions on `set + get` | `update(key, fn)` is the documented atomic op. Guide page covers the patterns. |
| Typed `stateMapping` generic propagation breaks an existing call site | Default generic is `NodeStateBase`; existing code typechecks unchanged. |
| Bundling typed-stateMapping with Store work means a Store design slip blocks the smaller release | User explicitly accepted this trade-off. |

## Follow-ups beyond v0.11

- `Path<TParent>` index-signature behavior: today's `Path<T>` falls back to
  `string` when `T` has an index signature. A stricter implementation that
  enumerates the union case is possible but type-heavy; defer to v0.12 if
  consumers ask.
- `RemoteStore` reference implementation. The contract ships in v0.11; an
  HTTP/WS reference plugin is deferred until consumers materialize.
- Cross-process atomicity for `EventLogStore` file mode. Today the file is
  append-only with `O_APPEND` semantics from the OS — multiple processes
  writing the same file may interleave entries. Document or add advisory
  locking in a follow-on release.
