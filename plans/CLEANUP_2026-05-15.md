# Dagonizer Cleanup ā€” Binding Plan (2026-05-15)

Backbone-readiness cleanup. The four candidate consumers (nocturne, iridis,
ripperoni, squashage) cannot adopt `@noocodex/dagonizer` without the gaps
listed here. Plan is binding: no scope reduction, no deferral to future
versions, no "tracked separately for follow-up." If a phase reveals more work
than estimated, dispatch additional parallel agents ā€” do not cut.

## Operating principles

- **Class extension is the only extension mechanism.** Zero callbacks. Zero
  function-pass-in. Consumers extend classes or implement adapter contracts.
- **Domain modules with `noun.verb()` only.** No freestanding helpers. No
  `makeX` / `buildX` / `parseX` / `fromX` verbs in module scope. Static
  methods on a static class.
- **Interfaces colocate with classes.** Same file. Always.
- **Adapter contracts live at the root of `contracts/`.** Single source of
  truth ā€” no re-exports from sibling modules.
- **Schemas are the source of truth.** Every wire-shape entity has a
  `*Schema` value and a `FromSchema`-derived type.
- **All public types ship through submodule exports.** Consumers extend and
  compose; nothing extensible may be private.

### 1.4 Document the three-tier interface taxonomy (item 15)

## Phase 3 ā€” Service container

### 3.1 Generic `NodeContextInterface<TServices>` (item 10)

Add a `services` slot to `NodeContextInterface`:

```ts
export interface NodeContextInterface<TServices = void> {
  readonly signal: AbortSignal;
  readonly dagName: string;
  readonly nodeName: string;
  readonly services: TServices;
}
```

`Dagonizer<TState, TServices = void>` accepts services at construction time:

```ts
new Dagonizer<MyState>();                       // TServices = void
new Dagonizer<MyState, MyServices>(services);   // TServices = MyServices
```

`buildContext` (`Dagonizer.ts:297-307`) populates `services` from the constructor.

`NodeInterface<TState, TOutput, TServices = void>` propagates the parameter
through `execute(state, context)`.

Subclassed dispatchers wire their own services. No DI container, no provider
scope ā€” the consumer's responsibility is to construct the services bag and
hand it in once.

`NodeContextSchema` does not change (services are runtime, not wire).
